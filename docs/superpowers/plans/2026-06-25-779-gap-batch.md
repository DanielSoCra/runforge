### 4b. phases.ts — detect-settled signal + post-gate assertion + reconcile

1. Add an optional `onDetectSettled?: () => void` parameter to
   `createPhaseHandlers` (append to the signature so existing callers stay valid).
2. In the `detect` handler `finally` (~492), **call `releaseRepoGitLock()` FIRST,
   THEN `onDetectSettled?.()`** (order is load-bearing — codex plan-review): if

### 4c. daemon.ts — centralized detect-scoped dispatch gate (ALL in-process entrants)
1. Daemon-scoped registry near the poll state: `const detectInFlight = new Set<string>()`
   keyed by repo (reuse the existing repo key, e.g. `${owner}/${name}` or `repoId`).
2. **Gate placement — check/set at the CLAIM/COMMIT point, NOT the dispatch site**
   (codex plan-review CRITICAL). For fresh work, `claimWork` + `activeIssues.add`
   + `activeRuns++` happen in the CLAIM loop (~1219–1223) BEFORE the dispatch site
   (~1233); skipping at dispatch would STRAND an already-claimed `in-progress`
   issue. So:
   - In the fresh-work claim loop (~1197–1226), BEFORE `claimWork` (~1219):
     ```ts
     const repoKey = `${owner}/${name}`;
     const entersAtDetect = getStartPhase(selectVariant(request)) === 'detect';
     if (entersAtDetect && detectInFlight.has(repoKey)) continue; // skip claiming a detect-run while one is in flight
     ```
   - AFTER a successful claim + `activeRuns++` (~1223): `if (entersAtDetect) detectInFlight.add(repoKey);`
     and create the per-run release, carried to the dispatch loop (e.g. a
     `Map<issueNumber, () => void>`):
     ```ts
     let gateReleased = false;
     const releaseDetectGateOnce = () => { if (!gateReleased) { gateReleased = true; if (entersAtDetect) detectInFlight.delete(repoKey); } };
     ```
   - Bug-fix (~1284) and feature-pipeline (~1344) claim+dispatch sites: same — gate
     check before their claim, `add` on commit.
   - Crash-resumption (~1567–1660): no `claimWork` — `add(repoKey)` at `activeRuns++`
     (~1587) iff `run.phase === 'detect'`. **It is a ONE-SHOT startup pass
     (daemon.ts ~1559) — no retry tick (codex plan-review):** do NOT skip-and-drop
     a gated crash-resume detect run. If `detectInFlight.has(repoKey)`, QUEUE/DEFER
     the launch (per-repo FIFO; fire the next from `releaseDetectGateOnce`) or
     launch same-repo crash-resume detect runs serially — every incomplete detect
     run must launch exactly once.
3. Thread the per-run `releaseDetectGateOnce` (created at the commit point, step 2,
   carried via the `Map<issueNumber, release>`) into `processWorkRequest` →
   `createPhaseHandlers` as `onDetectSettled` (non-website branch ~2651). Website
   branch: omit (start phase `init`; never gated). Look up / pass the release for
   each request at the dispatch site (~1233) from the Map.
4. Backstop release that covers SETUP THROWS, per path (codex round-2 #3):
   - **fresh-work:** attach the request's `releaseDetectGateOnce()` to the existing
     `processWorkRequest(...).catch(...).finally(...)` (~1260). Because
     `processWorkRequest` is an async fn, a throw in its setup (saveRunState/
     insertRun/token/readAgencyConfig/handler-build) rejects the promise → the
     `.finally` runs. Sufficient here.
   - **crash-resumption (~1567–1660):** the setup (`activeRuns++`,
     `readAgencyConfig`, handler build, ~1595–1643) is INLINE before `runPipeline`
     (~1650). A throw there escapes before `runPipeline(...).finally` exists →
     gate leak. Correct shape (codex round-3 CRITICAL — do NOT use an outer
     `finally`; `runPipeline` is async/fire-and-forget so an outer `finally` would
     release synchronously BEFORE detect runs):
     ```ts
     try {
       ...inline setup...
       runPipeline(...).then(...).catch(...).finally(releaseDetectGateOnce); // run-level backstop
     } catch (setupErr) {
       releaseDetectGateOnce();  // ONLY on a synchronous setup throw
       ...handle...
     }
     ```
     Early release still comes from `onDetectSettled` (detect's `finally`).
     Per-run idempotency keeps the double-release safe.
5. **Gate the crash-resumption path** (`findIncompleteRuns` loop ~1567–1660): gate
   iff `run.phase === 'detect'` (the resumed entry phase), with the setup-throw
   try/catch from step 4. Crash-resumption builds handlers INLINE via
   `createPhaseHandlers` (~1624) — NOT through `processWorkRequest` — so pass
   `onDetectSettled: releaseDetectGateOnce` into THAT inline `createPhaseHandlers`
   call too, giving the same EARLY release (detect's `finally`) plus the
   `runPipeline(...).finally` backstop. Do NOT gate `reenterPipeline` (~2217,
   parked-decision resume) — it resets `run.phase` to post-detect phases and never
   re-runs detect.
