# P4.2 Pre-Approved Earn-In Promotion Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build the pre-approved earn-in promotion mechanism — a pure floor evaluator composed over the existing bar, a separate mint step at the integrate seam that auto-widens autonomy through the registry's reversible write, the debut gate for a deployment's first unattended merge, and the automatic demote-on-red reversal — so a deployment's lane can widen off its own proven track record without a per-promotion Operator decision, behind non-configurable fail-closed floors.

**Architecture:** Realizes ARCH-AC-EARN-IN / STACK-AC-EARN-IN over FUNC-AC-MERGE-DECISION v2.3 + FUNC-AC-FLEET. Pure decisions (`evaluatePromotion`, `derivePromotionTrackRecord`, `planMint`, `isDebut`, `isRedEvent`) live in exhaustively-testable functions; I/O and state mutation (the mint's `recordWidening`, the outcome-ledger append, the release-ledger reads) sit at the integrate edge in `phases.ts`. `decideMerge` stays **pure and untouched** — the mint is a sibling state-mutation step that runs *before* `decideMerge` reads autonomy, so the very next decision reads the freshly-widened state. Every earn-in widening is recorded through `recordWidening` and is level-wide-reversible by the demote-on-red trigger.

**Tech Stack:** TypeScript (daemon Control Plane), Zod (config schema), Drizzle/Postgres (release-ledger), JSON-file stores (autonomy state, metrics ledgers), Vitest.

## Global Constraints

Copied verbatim from the specs. Every task's requirements implicitly include this section.

- **Floors are a platform constant, non-configurable.** `EARN_IN_FLOORS = { minCleanMerges: 10, recencyWindowDays: 30, redWindowDays: 30 }` — PROVISIONAL pending the Operator's ruling (bridge #104). A single frozen module constant tied to the bridge item by one comment; the Operator's decision changes the numbers, never the mechanism. No pack, profile, or lane field parses into it.
- **The bar is necessary but not sufficient.** Both a missing bar and a declared bar that sits below any numeric floor (`bar.cleanMerges < minCleanMerges` or `bar.bounceFreeDays < recencyWindowDays`) **fail closed — they never auto-widen** (the safety-identical property from FUNC-AC-MERGE-DECISION v2.3). They fail closed via *distinct outcome kinds*, matching ARCH-AC-EARN-IN Error Handling (`earn-in.md:89`/`:91`): a **missing** bar (or unestablished record) → `not-eligible` (the bar was never met); a **weak** bar the record clears but that sits below a floor → `raise-decision` carrying `bar-clean-merges-below-floor`/`bar-recency-below-floor` in `failedFloors`. Neither yields `auto-widen`. "Fail closed identically" means identical *safety* (no auto-widen), not an identical outcome tag.
- **Recency is freshness, not streak length.** The recency floor requires **at least `minCleanMerges` clean merges whose timestamp is `>= now - recencyWindowDays`** — NOT merely that the most-recent clean merge is recent. A dormant lane with high cumulative `cleanMerges` all months old fails the floor. `now` is passed in; the pure functions never read a clock.
- **Auto-widen ONLY when ALL hold:** bar met, `preApprovedEarnIn` present, every floor clears (bar ≥ every numeric floor, recency count in window, no red in window, scope holding, verifier-gated, reversible), the level is autonomous-eligible (green/yellow — never orange/red), compliance is not forcing, and the (deployment, level, lane) is currently human-gated. Any indeterminate input collapses to `not-eligible` or `raise-decision`, never `auto-widen`.
- **Never crosses an always-escalate boundary.** The mint is never taken for orange, red, compliance-forced, or a lane without a usable falsifying verifier. The guard precedes the `recordWidening` call, never follows it.
- **`decideMerge` stays pure.** Do not thread earn-in into `merge-decision/decide.ts`. The mint composes as a separate step ahead of the decision.
- **The mint is idempotent.** It fires only at the `human-gated → widened` crossing (gate on `currentlyHumanGated`); a second integrate pass is a no-op.
- **Fail-closed persistence.** Treat a non-`ok` `recordWidening` outcome as "not widened"; never report an auto-promotion that did not durably record (an unrecorded widening is neither reversible nor explainable).
- **Demote-on-red is level-wide + sets the marker.** `recordWidening(id, class, 'human-gated', auth, ts)` with NO `lane` argument (clears the class's lane widenings + appends per-lane revocations), AND the same red event sets the red-window marker. Any doubt about whether a signal is a red event resolves to demoting, never to retaining autonomy.
- **The debut is derived from recorded state, never a label.** `isDebut(id) = !history.some(r => r.next === 'widened')`. The first-ever widening of ANY authorization is the debut; any prior widening (operator-grant OR earlier earn-in-policy) means the crossing was already witnessed. Renaming/cloning/re-bundling cannot evade it.
- **No new Operator approval beat.** The debut is folded into the production-release approval the deployment already requires (a third option `approve-with-debut`), offered only until the deployment has a prior approved release. A deployment with no release path fails closed to a per-event decision but is never trapped closed forever.
- **Specs stay `draft`.** STACK-AC-EARN-IN and ARCH-AC-EARN-IN are promoted to a higher status only at the Phase-8 live proof, not by this plan.

## Boundary & Ownership Decisions

These are the load-bearing structural calls this plan makes. Read before implementing.

1. **Net-new earn-in code lives under `packages/daemon/src/control-plane/earn-in/`** (a new directory the STACK-AC-EARN-IN node owns), NOT inside `lane-engine/`. Earn-in composes over the Lane Engine; it does not re-own any lane-engine decision. The one exception is the **raw outcome ledger** (Task 3), placed under `lane-engine/` because ARCH-AC-EARN-IN's boundary assigns "the recorded track record and outcome stream" to the Lane Engine — earn-in owns only the *floor-relevant derivation* over it.

2. **OPEN QUESTION for the team lead (recommendation given, not blocking):** The specs assume a "recorded track record + outcome stream" the earn-in derivation reads, but **none exists today** — `evaluateEarnIn`/`LaneTrackRecord` are untethered scaffolding (never called outside their own test), and there is zero bounce/red recording anywhere in the daemon. This plan therefore **builds a minimal per-(deployment, lane) outcome ledger** (Task 3) + emission at the integrate merge/park seams + the demote trigger, so the mechanism is buildable and Phase-8-provable end-to-end. *Recommendation:* keep the raw ledger in this plan under `lane-engine/` as above. *Alternative the team lead may prefer:* split the outcome ledger into a separate STACK-AC-LANE-ENGINE plan and have this plan consume a `PromotionTrackRecord` input contract only (leaving earn-in unprovable-live until that lands). Proceeding with the recommendation unless redirected.

3. **`AutonomyAuthorization` gains a `demote-on-red` kind** (Task 6). An automatic demote has no operator and no policy, so neither existing kind (`operator-grant`, `earn-in-policy`) honestly represents it; `isValidAuthorization` must accept the new kind. This is a small, spec-driven addition to a STACK-AC-DEPLOYMENT-REGISTRY-owned type — flagged, and its traceability recorded in Task 11.

4. **Two net-new release-ledger reader methods** (Task 7): `hasPriorApprovedRelease(deployment)` (to decide whether to offer the debut option) and `hasDebutAuthorization(deployment)` (the debut gate's read). Both scan the deployment's `decision` events; there is no existing cross-release reader for them.

5. **The withhold path reuses the existing escalate → Operator-decision flow.** When the debut gate withholds an auto-mint, no widening is recorded, so `autonomyWidened` reads human-gated and `decideMerge` escalates `autonomy-not-widened` — surfacing the change to the Operator as the per-event decision. Earn-in's responsibility ends at *withholding the auto-mint*; the Operator's grant recording an `operator-grant` widening is the existing per-event grant mechanism (not net-new here). Once any widened record exists, `isDebut` returns `false` and future promotions mint — so a no-release-path deployment is never trapped.

**Interfaces:**
- Consumes: `LaneOutcome` (Task 2), `WideningRecord` (`deployment-registry/types.ts`), `EarnInFloors` (Task 1), `LaneTrackRecord` (`lane-engine/types.ts`).
- Produces:
  - `interface PromotionTrackRecord { bar: LaneTrackRecord; cleanMergesInWindow: number; redEventInWindow: boolean }` — the "derived Promotion Track Record" the L2 API names as an input to `evaluatePromotion`. `bar` = `{ cleanMerges, bounceFreeDays }` for the Lane Engine's bar predicate; `cleanMergesInWindow` + `redEventInWindow` for the floors.
  - `function derivePromotionTrackRecord(outcomes: LaneOutcome[], autonomyHistory: WideningRecord[], now: number, floors: EarnInFloors): PromotionTrackRecord`

**Derivation rules (all read recorded timestamps only; `now` passed in):**
- `bar.cleanMerges` = count of `kind === 'clean-merge'` outcomes for the lane (cumulative — includes Operator-approved pre-autonomy merges; NEVER filtered to autonomous merges).
- `bar.bounceFreeDays` = whole days between `now` and the most recent `kind === 'bounce'` outcome (or between `now` and the earliest recorded outcome if no bounce). Guard: if there are zero outcomes, `bounceFreeDays = 0`.
- `cleanMergesInWindow` = count of `kind === 'clean-merge'` outcomes whose `ts >= now - recencyWindowDays*86400000`. This is the recency floor's numerator — the FULL count in window, not "most-recent within window".
- `redEventInWindow` = `true` iff (any `kind === 'red'` outcome with `ts >= now - redWindowDays*86400000`) OR (any `autonomyHistory` record with `next === 'human-gated'` and `recordedAt >= now - redWindowDays*86400000`) — the demote-on-red `WideningRecord` is a red signal too.

## Task 4: `evaluatePromotion` + `floorsFailed` (pure floor evaluator over the bar)

**Files:**
- Create: `packages/daemon/src/control-plane/earn-in/promotion-policy.ts`
- Modify: `packages/daemon/src/control-plane/earn-in/types.ts` (`PromotionInput`, `PromotionResult`)
- Test: `packages/daemon/src/control-plane/earn-in/promotion-policy.test.ts`

**Interfaces:**
- Consumes: `evaluateEarnIn` (`lane-engine/earn-in.ts`), `EARN_IN_FLOORS`/`FloorName`/`FLOOR_NAMES` (Task 1), `PromotionTrackRecord` (Task 3), `EarnInPolicy`/`LaneTrackRecord` (`lane-engine/types.ts`).
- Produces:
  - `interface PromotionInput { record: PromotionTrackRecord; bar: EarnInPolicy | undefined; preApproved?: { enabled: boolean; policyRef: string }; verifierFalsifying: boolean; scopeHolding: boolean }`
  - `type PromotionResult = { kind: 'not-eligible' } | { kind: 'raise-decision'; failedFloors: FloorName[] } | { kind: 'auto-widen'; clearedFloors: FloorName[]; evidence: PromotionTrackRecord; policyRef: string }`
  - `function evaluatePromotion(i: PromotionInput): PromotionResult`
  - `function floorsFailed(i: PromotionInput): FloorName[]` (exported for the acceptance gate)

**Design (faithful refinement of the L3 example):** the L3 sketch is `evaluatePromotion(i, now)` with derivation inline; this plan **splits** derivation (Task 3, owns `now`) from evaluation (pure comparison over the already-derived `record`), matching the L2 API's "derived Promotion Track Record" *input*. `evaluatePromotion` takes no `now`. Composition:
1. `if (evaluateEarnIn(i.record.bar, i.bar).kind !== 'eligible-for-promotion') return { kind: 'not-eligible' }` — the bar short-circuits; earn-in never forks it. A **missing** bar (`i.bar === undefined`) makes `evaluateEarnIn` return `not-eligible`, so `evaluatePromotion` returns `not-eligible` (fail closed — never auto-widen). A **present-but-weak** bar the record clears reaches step 2, where `bar-clean-merges-below-floor`/`bar-recency-below-floor` push it to `raise-decision` (also fail closed). Both are safety-identical (no auto-widen); the outcome tags differ by design (see the Global Constraint).
2. `const failed = floorsFailed(i)` — ALL floor failures:
   - `bar-clean-merges-below-floor` iff `i.bar === undefined || i.bar.cleanMerges < EARN_IN_FLOORS.minCleanMerges`
   - `bar-recency-below-floor` iff `i.bar === undefined || i.bar.bounceFreeDays < EARN_IN_FLOORS.recencyWindowDays`
   - `insufficient-recent-clean-merges` iff `i.record.cleanMergesInWindow < EARN_IN_FLOORS.minCleanMerges`
   - `red-in-window` iff `i.record.redEventInWindow === true`
   - `scope-not-holding` iff `i.scopeHolding !== true`
   - `verifier-not-gated` iff `i.verifierFalsifying !== true`
   - `reversible` — NEVER added (satisfied by construction: the mint records through `recordWidening`); always in `clearedFloors`.
3. `const eligible = i.preApproved !== undefined && i.preApproved.enabled === true && failed.length === 0`. A declared-but-**disabled** policy (`enabled: false`) is inert — falls through to `raise-decision`, the v2 default (this reifies ARCH's "enabled flag"; the **verifier-gated linkage** the ARCH data model names is enforced structurally — the lane's own `verifier` declaration + the mint's `verifierFalsifying` guard + the `verifier-not-gated` floor — never as a self-asserted policy field, because a policy cannot declare a property of the lane).
4. `return eligible ? { kind: 'auto-widen', clearedFloors: FLOOR_NAMES, evidence: i.record, policyRef: i.preApproved.policyRef } : { kind: 'raise-decision', failedFloors: failed }`.

`floorsFailed` is complete audit evidence — the numeric bar-below-floor entries sit alongside the structural floors, so a weak declared bar surfaces in `failedFloors`, never as a silent non-match.

- [ ] **Step 1: Write the failing test** — cases (build `PromotionInput` literals):
  - Bar not met (`record.bar.cleanMerges < i.bar.cleanMerges`) → `not-eligible`.
  - **Missing bar:** `i.bar === undefined`, record strong, `preApproved` present + enabled → `not-eligible` (fail closed; distinct tag from the weak-bar case but equally never `auto-widen`).
  - Bar met, `preApproved` undefined, all floors clear → `raise-decision` with `failedFloors: []` (v2 default).
  - Bar met, `preApproved` present + `enabled: true`, all floors clear → `auto-widen` with `policyRef` and `clearedFloors === FLOOR_NAMES`.
  - **Disabled policy:** bar met, `preApproved` present but `enabled: false`, all floors clear → `raise-decision` (never `auto-widen`).
  - **Weak bar:** `bar.cleanMerges = 3` (`< 10`), record otherwise strong, `preApproved` present + enabled → `raise-decision` with `failedFloors` containing `bar-clean-merges-below-floor` (NOT `auto-widen`).
  - `redEventInWindow: true` with everything else clear + `preApproved` → `raise-decision` with `red-in-window`.
  - `cleanMergesInWindow: 5` (`< 10`) but `bar.cleanMerges = 12` → `raise-decision` with `insufficient-recent-clean-merges`.
  - `verifierFalsifying: false` → `verifier-not-gated` in `failedFloors`; `scopeHolding: false` → `scope-not-holding`.
  - **No input path yields `auto-widen` while any `failedFloors` entry is present** (property test over a small matrix).

## Immovable Acceptance-Gate SPEC (CONDUCTOR's RED-gate input)

The gate is a set of tests the implementation must make pass and must never edit. Grouped by safety property; each bullet is a concrete falsifiable assertion. **[PURE]** = no DB (pure/filesystem); **[REAL_PG]** = needs real Postgres (release-ledger / release executor). The registry autonomy store is a JSON file, so registry assertions are **[PURE]**.

**A. Auto-widen only when ALL conditions hold — `evaluatePromotion` [PURE]**
1. Bar not met → `not-eligible` (no `raise-decision`, no `auto-widen`).
2. Bar met + `preApproved` present + `enabled: true` + every floor clears → `auto-widen` carrying the exact `policyRef` and `clearedFloors === FLOOR_NAMES`.
3. Bar met + `preApproved` **absent** → `raise-decision` (v2 default), never `auto-widen`, even with all floors clear.
3b. Bar met + `preApproved` present but **`enabled: false`** + every floor clears → `raise-decision`, never `auto-widen` (a declared-but-disabled policy is inert).
4. Property: over a matrix toggling each of {preApproved present, `enabled`, each floor}, `auto-widen` occurs **iff** `preApproved` present AND `enabled === true` AND `failedFloors` empty — no other combination yields `auto-widen`.

**B. A missing bar and a bar below a floor both fail closed — `evaluatePromotion` [PURE]**
