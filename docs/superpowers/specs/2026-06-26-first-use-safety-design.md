# First-Use Safety Hardening — Design Spec

**Date:** 2026-06-26
**Topic:** `first-use-safety`
**Status:** draft (codex spec-review pending)
**Base:** `origin/main` @ `a884135`
**Author:** Claude (conductor/planner), adjudicated with Codex GPT-5.5

## Goal

Make runforge usable for **attended first-real-use on a single-writer repo**. A verified blocker audit found three things stand between the proven demo and real use:

1. **The operator-approval surface is dead in the default config.** With `RUNFORGE_DECISION_INDEX_ENABLED` OFF, every `escalate`/`hold` merge decision returns `'failure'` instead of parking for operator approval — and a merge-governed deployment can boot blind with no warning. (Already fired in demo run #2.)
2. **Unattended ≠ visible.** Alerting is a silent no-op when no webhook is configured; the daemon can auto-pause/escalate into the void.
3. **A wedged run looks healthy forever.** `/health` is a hardcoded constant; a non-settling orchestration await leaks a concurrency slot with no detection.

This spec delivers the **in-bounds hardening** that closes these (no L0/L1 spec edits — every change realizes an *existing* L1 mandate), split into **two PRs**. One genuinely-new capability (an out-of-process dead-man's-switch) is identified and **deferred** to an Operator L1 decision; it is *not* built here.

## Chosen Design

### PR1 — Decision-index approval surface (the hard-blocker; ships first)

**A1 — Boot guard (hard refuse), covering BOTH index-unavailable AND registration failure.** Immediately after deployment registration (`daemon.ts:~366-376`), add a guard for a configured deployment (`config.deployment !== undefined`) that refuses boot via the existing boot-error path (`return err(...)`, mirroring the sanitizer fail-closed at `daemon.ts:384-400`) when **either**:
- (i) `!decisionManager.isAvailable()` — the approval surface is dead; **or**
- (ii) `deploymentRegistry.register(...)` **failed** — today this only `console.error`s (`daemon.ts:371-375`) and lets the daemon run with an empty registry, surfacing the failure later at integrate (Codex IMPORTANT 4). A1 promotes it to a boot error for a *configured* deployment.

The error message names the underlying reason in operator-readable form (FUNC-AC-SAFETY "underlying cause … observable"), distinguishing: **index disabled** (`!isEnabled()` → "set `RUNFORGE_DECISION_INDEX_ENABLED=1`"), **index enabled-but-unreachable** (`isEnabled() && !isAvailable()` → "decision index unreachable: <reason>"), and **registration rejected** (→ the registry's rejection reason).
- *Predicate = `config.deployment !== undefined`* (Codex Q2): a deployment profile *is* the merge-governance boundary; even all-permissive lanes cannot remove no-verification / unknown-risk / compliance / orange-red / first-unattended / phase-change escalations.
- *Severity = hard refuse* (Codex Q1): the demo "works" green-only by path luck; a governed profile with no approval surface converts a required park into silent runtime failure. Refusing forces the operator to either enable the index or drop `deployment`. **This intentionally requires the demo to run index-ON** — which is exactly the A2 proof.

**A2 — Prove the round-trip + close the test gap.** (Codex IMPORTANT 3: PGlite **cannot** back the real `IndexWriter` — the harness comment at `decision-escalation/__fixtures__/pg-test-harness.ts:5-8` states the single-writer + read-only-session guarantees "cannot be modelled by an in-process backend". So a structural fake of the *writer* is out; instead seam at the *ledger contract* level.)
- **CI-default test (no Postgres):** introduce the seam at the call-site boundary. Enumerate the **exact** interfaces from the real callers (Codex round-2 IMPORTANT 4 — derive from `daemon.ts` boot `~336-376`, `resumeParkedRuns`/`resumeIntegrateParkedRun` `~2099/2449-2630`, the integrate park/publish at `phases.ts:2217-2252`, and `reconcile.ts`):
  - **`DecisionManagerLike`** — the real manager surface the boot/resume callers touch (verified against `daemon.ts`): `init()` (`daemon.ts:341`), `isEnabled()`, `isAvailable()`, `ledger()`, `protectedStore()` (`daemon.ts:354`), `close()` (shutdown), and the new runtime-degraded marker accessors from B4. Note: `bootReconcile(manager)` is a **free function** (`daemon.ts:344`), not a manager method — it needs only `isEnabled()` + `ledger().reconcile()` (`reconcile.ts`). If the test builds the full control server, also include `reader()`/`revealProtected()` per `startDaemon` typing.
  - **`DecisionLedgerLike`**: `raise()`, `notify()`, `answer()`, `statusOf()`, `advanceToResumed()`, `reconcile()`, plus any tick-maintenance method (e.g. `markOverdue`) the chosen entry point touches. **Derive the exact set from the test's actual entry points** (`resumeParkedRuns`/`resumeIntegrateParkedRun` + the integrate phase handler) — do not guess a subset; a missing method fails compilation, which is the desired forcing function.
  - The **state-machine fake** enforces: `raise`→`raised`; `notify`→`notified`; `answer` **rejects a choice not in the offered options**, applies **answer-once** (identical re-answer is a no-op; a *conflicting* re-answer is rejected; a terminal/`resumed` row is unchanged), →`answered`; `advanceToResumed`→`resumed`.
  - Wire it (the `DecisionManagerLike` returns the fake from `ledger()` + `isAvailable()===true`) into the *joined* chain: integrate park (`phases.ts`) → mock publisher → synthesized `**DecisionResponse**` approve comment → `parseCockpitAnswer` → `resumeIntegrateParkedRun` → integrate override with a **mock `integrateToStaging`** asserting the held merge is **executed** on approve; a second case asserts **reject** routes back to `implement` with feedback and **no** merge; a third asserts a conflicting double-answer is rejected (answer-once).
- **Real-Postgres test (extend, don't replace):** extend the existing `daemon.test.ts:4283` `describe.skipIf(!REAL_PG)` integrate-resume test so it continues *through* the integrate override and asserts the merge step is reached (today it stops at "resumed"). This runs in CI where `RUNFORGE_TEST_DATABASE_URL` is set, covering the **real** writer/outbox/CAS semantics the fake intentionally abstracts.
- **E2E (Phase 9):** flip `RUNFORGE_DECISION_INDEX_ENABLED=1` against the demo Postgres, drive one real `escalate` change end-to-end, post the approve DecisionResponse (via `decision-api.ts answerDecision` or a direct comment), confirm the run resumes and merges. (See End-to-end verification.)

### PR2 — Unattended safety net (ships second)

**B1 — Alerting wired + non-silent empty channel.** Keep `notify()`'s per-URL behavior; make the **empty-channel case observable** at the pause sites (a structured `console.warn` when an auto-pause/escalation fires with zero channels) so the no-op is never silent. No change to SSRF logic.

**B2 — Alert-channel *degraded* (warn, NOT refuse) — Codex CRITICAL 1.** Add `hasConfiguredAlertChannel(config): boolean` (Codex Q3 — an abstraction, not raw `webhooks.length`). At boot, if the daemon is governed (`config.deployment !== undefined`, same predicate as A1) **and** `!hasConfiguredAlertChannel(config)`: emit a **loud startup warning** and set a persistent **`degraded:true`** flag that `/health` reports. Do **not** refuse boot. Rationale: existing L1 mandates "don't run silently" + "make degraded state observable" — which warn+degraded fully satisfies — but **no L1 says an alert channel is *required*** for a governed deployment, so a hard boot-refuse (no opt-out) would be net-new operator policy. The hard-refuse variant (`B2-strict`) is deferred to an Operator L1 decision. Non-governed local runs are unaffected.

**B4 — Truthful `/health` (real server) — Codex IMPORTANT 2 + MINOR 1.** Thread a real state provider (`getStatus()` + heartbeat age + watchdog state + the B2 alert-channel-degraded flag) into the **real** server's `/health` handler (`server.ts:71-75`). The **degraded-boot** server (`degraded-server.ts:36-37`) already returns `degraded:true` during startup and is unchanged; document the handoff (degraded server serves until the real server binds post-recovery). Status-code mapping (Codex Q4):
- **503** (unsafe / no-progress): `consecutiveStuckCount >= maxConsecutiveStuck`; watchdog detected tick-stall or run-stall past idle-timeout; a repo tick stale beyond N intervals while **not** paused/draining; a **safety** auto-pause (`pauseReason ∈ {budget, stuck, tick-error, runtime-source}`) persisting beyond grace; **a governed daemon whose decision index has failed at runtime** (see next bullet).
- **200 + `degraded:true`**: startup-degraded while still retrying; **operator/manual** pause (`pauseReason='manual'`); draining; transient alert-channel send failure; **governed-without-alert-channel** (B2).
- **200 + `ok:true`**: normal.
- **Decision-index state in `/health` (Codex IMPORTANT 2 + round-2 IMPORTANT 3) — governed-only, runtime-aware.** Index *disabled* never degrades a **non-governed** daemon (legacy integrate is normal there). But A1 only guarantees *boot-time* availability — a **governed** daemon's Postgres can die *after* boot, making every escalate-merge fail-closed (`phases.ts:2193-2200` / the publish catch at `:2247`) while `/health` would otherwise read OK. So: **PR1 adds a runtime-degraded marker on `DecisionIndexManager` with a governed-only marking policy** (Codex round-3 IMPORTANT 2) — a small wrapper that marks the index runtime-degraded on **any** failure of an approval-path ledger interaction for a governed deployment, covering **all** the real blocking sites, not just two catches: the `phases.ts:2193` floor miss; integrate publish `raise`/`notify` (`phases.ts:2247-2251`); resume `statusOf` (`daemon.ts:2515-2522`), `answer` (`daemon.ts:2560-2568`), and `advanceToResumed` (`daemon.ts:2620-2626`); and tick-loop reconcile/markOverdue for governed deployments (`daemon.ts:1293-1304`). The marker is **cleared only by a successful *governed* merge-decision op** — never by an unrelated non-governed or l2-gate read. **`/health` returns 503 for a governed daemon when the marker is set** (or `isEnabled() && !isAvailable()`). This keeps the runtime fail-closed state observable, which the whole approval design depends on.
- *Do not treat plain `paused` as unhealthy*: introduce/propagate **`pauseReason`** (`'manual' | 'budget' | 'stuck' | 'tick-error' | 'runtime-source'`) at the pause sites (`daemon.ts:1124/1188/1206/1237/1662`); any un-tagged pause defaults to the cautious (safety→503) interpretation.

**B5 — Work-loop watchdog: DETECT → safe-pause + notify + 503 (does NOT terminate the run) — Codex CRITICAL 2 + IMPORTANT 1/2.** The watchdog **detects** a stall using signals the daemon already has — **no force slot-release, no `activeRuns` decrement, no cancellation** (decrementing while the original promise is alive risks double-decrement, overlapping work, and a marked-stuck run still mutating GitHub/state — Codex CRITICAL 2). Two detectors:
- **Run-stall:** an `activeRuns>0` run whose `run.updatedAt` (bumped on every `saveRunState`, `state.ts:22`) has not advanced for **> idle-timeout**. `updatedAt` advances on phase transitions and durable progress saves (FUNC-AC-SAFETY "Progress recovery"), so a legitimately-long but *progressing* run keeps it fresh.
- **Tick-stall:** a repo whose poll started but never completed for **> idle-timeout** — a hung orchestration await between/around phases. `repo-manager.ts` exposes only the `pollInProgress` boolean (`:171`) with no start time, so B5 **adds `pollStartedAt` to the RepoEntry + a read-only poller-snapshot accessor** (MINOR 2) that the watchdog reads with an injected clock.

On detection: **self-pause** (`pauseReason='stuck'`, stops claiming *new* work — safe, no double-decrement) + **notify** + flip `/health` to **503**. **Honest scope (Codex IMPORTANT 1):** B5 does **not** recover the held slot. The 3h subprocess kill (`session-runtime/runtime.ts:204`) only releases the slot for an *in-worker subprocess* hang — for an **orchestration-level** hang (Octokit/git/pnpm/ledger, the real gap) there is no live subprocess, so `.finally()` may never run and the slot stays held. That is acceptable: the daemon has **self-paused** (claims no new work) and is **loudly degraded (503 + notify)**, so the "goes dark" gap is closed; clearing the held slot is **operator action** (restart) until the deferred `B5-cancel` adds real cancellation. **Idle-timeout default = subprocess-max (3h) + grace (15m)**, configurable downward for pilots — chosen so (a) *in-worker* long-but-progressing phases (bounded at 3h) never false-positive, (b) *orchestration* untimed awaits are caught after the threshold, (c) legitimately-long orchestration steps are minutes, never hours. **No session-heartbeat plumbing is added** (the subprocess-kill bound on in-worker time + the existing `updatedAt`/new `pollStartedAt` signals suffice — stated explicitly so the reviewer can check the reasoning).
- Add `process.on('uncaughtException')` / `('unhandledRejection')` handlers in `main.ts` that notify (if a channel exists) and exit gracefully; a crash-loop note for the plist (`KeepAlive=true`+`ThrottleInterval=30`) is documented (no restart-storm fix required for attended use).

## Rejected alternatives (Codex adjudications + spec-review)
- **A1 loud-warn-and-continue / config opt-out** — rejected (Q1): continues to convert required parks into silent failures; opt-out blesses a silent governed mode. (A1 IS hard-refuse because the index is structurally required for *any* governed change to satisfy "always reaches the Operator"; B2 is NOT, because no L1 requires an alert *channel* — that asymmetry is the whole CRITICAL-1 fix.)
- **A1 predicate = "profile has escalating lanes"** — rejected (Q2): false-negative trap; permissive lanes still escalate.
- **B2 hard-refuse on no alert channel** — rejected (spec-review CRITICAL 1): net-new operator policy not in any L1 → deferred as `B2-strict`. Shipped form is warn + `degraded:true`.
- **B2 gate on `autonomous`** — rejected (Q3): that flag is CLI-trust, not unattended-mode.
- **B4 treat all `paused` as 503 / factor in decision-index availability** — rejected (Q4 + IMPORTANT 2): manual pause is intentional; index-off is normal for non-governed mode.
- **B5 force slot-release / decrement `activeRuns` on timeout** — rejected (CRITICAL 2): double-decrement, overlapping work, and a stuck-marked run still mutating state. Shipped form detects + safe-pauses; the existing 3h subprocess kill remains the only termination path.
- **B5 add session-heartbeat plumbing** — not needed (IMPORTANT 1): the 3h subprocess-kill bound + existing `updatedAt`/`pollInProgress` signals suffice for the orchestration-stall gap.
- **B5 completion-only or pure wall-clock detection** — rejected (Q5): completion-only never fires on a hang; pure wall-clock kills legitimately-long runs. Progress-based idle timeout is the real detector.
- **A2 PGlite-backed real IndexWriter** — rejected (IMPORTANT 3 + harness comment): the real writer's single-writer/read-only-session semantics can't run in-process. Use a `DecisionLedgerLike` seam + lifecycle fake (CI-default) and extend the real-PG test (CI-with-PG).
- **One combined PR** — rejected (Q6): two PRs let the hard-blocker ship without waiting on health/watchdog tuning.

## End-to-end verification design (the headline proof)
On the demo deployment (`~/code/runforge-demo-runtime`, Postgres `runforge_demo`, target `cause-driven-tasks`):
1. **PR1:** set `RUNFORGE_DECISION_INDEX_ENABLED=1` + `RUNFORGE_DATABASE_URL`; boot — confirm the daemon **starts** (no longer refused, since the index is now available) and that with the index OFF it **refuses** (negative test). Seed a change that classifies **yellow/escalate** (e.g. touches a `riskPathMap` path like `package.json`/`tsconfig`/`.github`). Confirm: the run **parks** at integrate; a `decision-request`-labelled issue carries the embedded `DecisionRequest` block; post an **approve** `DecisionResponse` (via `decision-api.ts answerDecision` or comment); confirm the run **resumes and merges** to `main`. Repeat once with **reject** → routes back to implement. Capture the issue URL, the block JSON, and the merge SHA.
2. **PR2:** boot a governed config with **no** webhook → confirm it **boots with a loud warning** and `curl /health` → **200 `degraded:true`** (B2, not a refuse). ## Risks
- **A1 boot-refusal breaks an existing governed deployment** running index-OFF (incl. the demo). *Mitigation:* intended fail-closed; the rollout flips the demo config to index-ON, and the message says exactly what to set. Prominent in the PR1 body. (B2 does **not** refuse — only A1's index/registration guard does.)
- **B5 detect-not-kill leaves the hung slot held** until the 3h subprocess kill fires (the run is not force-released). *Accepted residual:* the daemon self-pauses + alerts + reports 503, so it is visible and stops taking new work — the "goes dark" gap is closed even though the hung run isn't actively cancelled. Force-cancellation is the deferred `B5-cancel` follow-up. For attended first-use this is the correct, safe scope.
- **Watchdog false-positive on a legitimately-long run.** *Mitigation:* progress-based (`updatedAt`/`pollInProgress`, not wall-clock) + default idle-timeout above the 3h subprocess bound + injected clock in tests.
- **`/health` 503 flapping.** *Mitigation:* grace windows on pause/stale; only hard no-progress/stuck/safety-pause states return 503; manual pause is 200-degraded.
- **The round-trip ON path has a latent bug** (never run E2E). *Mitigation:* the seam+fake joined test catches logic bugs and the extended real-PG test catches writer-semantics bugs pre-merge; the demo E2E catches integration/migration bugs. Failure modes are fail-closed-visible (`phases.ts:2187-2200`).
- **`pauseReason` plumbing touches several pause sites** (`daemon.ts:1124/1188/1206/1237/1662`) — risk of missing one. *Mitigation:* default any un-tagged pause to the cautious (safety→503) interpretation.
