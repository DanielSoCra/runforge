# STACK-AC-AGENT-DISCIPLINE-PROMPTS - Prompt and Context Contracts (TypeScript)

## Pattern

**Execution contract block in session prompts.** Worker-oriented prompts include a dedicated execution contract section with assumptions, ambiguities, non-goals, expected artifacts, success criteria, and verification commands. The section is generated by the Implementation Coordinator, not reconstructed by the Worker.

**Task graph success criteria.** Coordinator output includes success criteria per unit alongside expected artifacts and verification command. Success criteria are short observable outcomes, not implementation steps.

**Scope ledger in final reports.** Worker and bug-fix prompts require a final scope ledger listing changed artifacts and the reason each changed artifact was necessary. This makes unrelated edits visible to review.

**Conservative ambiguity routing.** Prompt templates map unresolved ambiguity to `NEEDS_CONTEXT` when more specification context could decide it, and `BLOCKED` when the unit cannot proceed safely within its assignment.

## Key Decisions

**Prompt blocks use explicit delimiters.** Add an `<execution-contract>` block beside existing context blocks. This follows the existing prompt variable wrapper pattern and makes caller-provided context visible while treating it as data.

**Extend the task graph schema, not free-form context.** Add `successCriteria` to unit output and validation. Keeping it structured prevents the Coordinator from hiding success criteria in prose that later systems cannot check.

**Use existing `estimatedChangeSize`.** Do not add a separate complexity budget unless implementation proves the existing estimate is insufficient. The discipline layer can compare final diff size to `estimatedChangeSize`.

**Final report stays text-compatible.** Worker prompts should require labeled sections for `Assumptions`, `Scope Ledger`, and `Verification Evidence` without requiring a new session transport format.

**No new agent type.** Reuse Coordinator, Worker, Spec Implementer, and Bug Worker sessions. The discipline is a prompt and context contract layered onto existing sessions.

```typescript
switch (ambiguity.severity) {
  case 'needs-parent-context': return 'NEEDS_CONTEXT';
  case 'unsafe-to-decide': return 'BLOCKED';
}
```

## Gotchas

- Success criteria must describe observable behavior. "Update service code" is not a success criterion; "invalid input returns a validation error" is.
- Non-goals are part of the contract. If a Worker changes a non-goal area, quality review should treat it as scope drift unless the governing spec required it.
- Do not duplicate full specification content in the execution contract. The contract summarizes interpretation; the governing specs remain separately injected.
- Keep contract rendering deterministic so prompt-contract tests can compare expected placeholders and rendered sections.
- A Worker should not invent new success criteria after implementation to fit what was built. If criteria change, the report must call that out as a concern.
